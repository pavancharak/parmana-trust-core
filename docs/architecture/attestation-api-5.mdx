# Attestation API Specification (Part 5)

## Attestation as the Foundation of Machine-Readable Trust

Historically, trust has been human-readable.

Examples:

```text id="3a2z8q"
Approval Email

Signed PDF

Audit Report

Compliance Document
```

These artifacts require human interpretation.

Attestations are different.

Attestations are:

```text id="6zv9f1"
Machine Readable

Cryptographically Verifiable

Portable

Deterministic
```

Trust becomes executable.

---

## From Human Trust to Machine Trust

Traditional approval:

```text id="0vqk1w"
Manager Approves
       ↓
Email Sent
       ↓
Action Taken
```

Trust is implicit.

Parmana model:

```text id="0dh4ps"
Authority Evaluation
          ↓
Attestation
          ↓
Verification
          ↓
Execution
```

Trust becomes explicit.

---

## Why AI Systems Need Attestation

Human workflows rely on observation.

AI workflows require evidence.

Example:

```text id="tp7rkr"
AI Agent
     ↓
Requests Payment
```

The system must answer:

```text id="sq7h5s"
Who authorized this?

Under what policy?

Using what evidence?

Can the authorization be verified?
```

Attestations answer these questions.

---

## Authority Without Trust Is Insufficient

Example:

```text id="0q8b3s"
Decision
 ↓
Approve
```

This establishes authority.

But it does not establish:

```text id="bnx4kw"
Integrity

Authenticity

Non-Repudiation
```

Attestation establishes those properties.

---

## Trust Without Authority Is Dangerous

Example:

```text id="z0jz2s"
Valid Signature
```

does not imply:

```text id="vw9q0r"
Correct Authorization
```

A signed decision may still violate policy.

Therefore:

```text id="l0n6xv"
Authority
     +
Trust
```

are both required.

## Attestation Portability Principle

An attestation should be portable across:

```text id="1yovv0"
Applications

Organizations

Cloud Providers

Vendors

Databases
```

Trust must not depend on infrastructure.

Trust must depend on evidence.

## Vendor Independence

Attestations should remain verifiable even if:

```text id="6dtixh"
Parmana No Longer Exists
```

Verification should still be possible.

Evidence must outlive vendors.

This is a foundational design principle.

## Long-Term Evidence Preservation

Example:

```text id="2m95sl"

Decision Audited
```

## Receipts Depend on Attestations

Receipts are derived artifacts.

Example:

```text id="d2w4n9"
Attestation
      ↓
Verification
      ↓
Receipt
```

Receipts summarize verification.

Attestations contain the original trust evidence.

---

## Transparency Logs Depend on Attestations

Transparency systems anchor evidence.

Example:

```text id="2yd6ys"

      ↓
Hash
      ↓
Transparency Log
```

Transparency logs prove:

```text id="zj6m8s"
Existence

Ordering

Tamper Detection
```

They do not replace attestations.
