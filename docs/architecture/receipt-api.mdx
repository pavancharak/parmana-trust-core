# Receipt API Specification

## Status

Planned

Version:

```text
v1
```

Endpoint:

```http
POST /receipt
```

---

## Purpose

The Receipt API produces an audit-ready artifact from a successful verification.

Evaluation determines:

```text
What was allowed?
```

Attestation determines:

```text
What was signed?
```

Verification determines:

```text
Can it be trusted?
```

Receipt determines:

```text
What evidence proves trust?
```

A receipt is the canonical audit artifact of the Parmana Authority Runtime.

---

## Runtime Position

```text
POST /evaluate
        ↓
Decision
        ↓
POST /attest
        ↓
Attestation
        ↓
POST /verify
        ↓
Verification
        ↓
POST /receipt
        ↓
Receipt
```

---

## Responsibility

The Receipt API:

```text
Creates Audit Artifacts

Packages Verification Evidence

Captures Runtime Metadata

Creates Long-Term Evidence Objects

Produces Portable Audit Records
```

The Receipt API does not:

```text
Evaluate Policies

Sign Decisions

Verify Signatures

Execute Actions
```

---

## Why Receipts Exist

Verification results are transient.

Receipts are durable.

Verification answers:

```text
Can this be trusted?
```

Receipt answers:

```text
How can this trust determination be proven later?
```

## Receipt Contents

A receipt captures:

```text
Verification Outcome

Attestation Reference

Trust Profile

Verification Timestamp

Receipt Timestamp
```

This creates a portable audit record.

---

## Receipt Identity

Every receipt has a unique identifier.

Example:

```json
{
  "receiptId": "rcpt-001"
}
```

The receipt identifier becomes the canonical reference used by:

```text
Auditors

Regulators

Enterprise Systems

Transparency Logs
```

---

## Receipt Provenance

Receipts inherit provenance from:

```text
Task

Policy

Decision

Attestation

Verification
```

Receipt generation does not create new authority.

It records trust evidence.

---

## Receipt Lifecycle

```text
Verification
      ↓
Receipt
      ↓
Store
      ↓
Retrieve
      ↓
Audit
```

Receipts are intended for long-term retention.

## Failure

```json
{
  "error": "verification required"
}
```

Receipts can only be generated from successful verification.

Policies define authority.

Trusted signals provide evidence.

Evaluation produces decisions.

Attestation creates cryptographic evidence.

Verification validates trust evidence.

Receipts preserve verified trust evidence.

Parmana makes authority verifiable, auditable, and enforceable before execution.
