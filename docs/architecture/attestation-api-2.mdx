# Attestation API Specification (Part 2)

## Why Attestation Exists

Policy evaluation determines authority.

Authority evaluation alone is insufficient for trust.

Example:

```text
Decision
 ↓
approve
```

A consumer cannot answer:

```text
Who produced this decision?

When was it produced?

Which policy generated it?

Was it modified?
```

Attestation solves this problem.

---

## Trust Boundary

The decision boundary is:

```text
Policy
 ↓
Decision
```

The trust boundary is:

```text
Decision
 ↓
Attestation
```

Attestation converts evaluation output into cryptographic evidence.

---

## Attestation Ownership

Attestations are runtime-generated artifacts.

They are not customer-authored.

They are not manually created.

The runtime is responsible for:

```text
Attestation Generation
Signature Creation
Evidence Packaging
Trust Metadata
```

---

## Attestation Immutability

Attestations are immutable.

After signing:

```text
Decision
Policy Reference
Timestamp
Signatures
```

must never change.

Modification invalidates trust.

---

## Canonical Attestation Structure

```json
{
  "attestationId": "att-123",

  "taskId": "payment.release",

  "policyId": "payment-release",

  "policyVersion": "1.0.0",

  "decision": {},

  "createdAt": "...",

  "signatures": []
}
```

The attestation contains sufficient information to independently verify the decision.

---

## Policy Provenance

Every attestation references the exact policy used.

Example:

```json
{
  "policyId": "payment-release",
  "policyVersion": "1.0.0"
}
```

This ensures:

```text
Reproducibility
Traceability
Auditability
```

Future policy changes cannot alter historical attestations.

---

## Task Provenance

Every attestation references the originating task.

Example:

```json
{
  "taskId": "payment.release"
}
```

This provides business context.

Without task provenance:

```text
Decision Exists
```

With task provenance:

```text
Decision Exists
For Which Business Action?
```

---

## Decision Provenance

Attestations preserve the exact decision.

Example:

```json
{
  "action": "approve",
  "requires_override": false,
  "reason": "payment_authorized"
}
```

The attestation proves that this specific decision was produced.

---

## Timestamp Provenance

Every attestation contains creation time.

Example:

```json
{
  "createdAt": "2026-06-17T12:00:00Z"
}
```

This establishes:

```text
Decision History
Chronology
Audit Trails
```

---

## Cryptographic Integrity

Attestation signing provides integrity.

Example:

```text
Attestation
      ↓
Hash
      ↓
Signature
```

Verification proves:

```text
Not Modified
```

after signing.

---

## Multi-Signature Model

Attestations support multiple signatures.

Example:

```json
{
  "signatures": [
    {
      "algorithm": "ed25519"
    },
    {
      "algorithm": "ecdsa-p256"
    }
  ]
}
```

Benefits:

```text
Threshold Trust

Multi-Party Approval

Federated Governance

Post-Quantum Migration
```

```text
Organization A
```

Example:

```text
Decision

Org B Signature
```

## Canonical Parmana Statement

Organizations decide what to trust.

Policies define what is required.

Trusted signals provide evidence.

Parmana evaluates those trusted signals against policy before execution.

Attestation creates cryptographic evidence of that authority decision.

Verification determines whether that evidence can be trusted.

Parmana makes authority verifiable and enforceable before execution.
