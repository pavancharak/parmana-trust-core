\# Lineage Model



\## Status



Canonical



\## Purpose



The Parmana Lineage Model defines the identifiers and relationships that connect trust-chain artifacts from authority evaluation through execution.



The lineage model provides:



\* Traceability

\* Auditability

\* Evidence preservation

\* Execution accountability

\* Cross-system correlation



The lineage model is the foundation of the Execution Trust Chain.



\---



\# Core Principle



Every trust-chain artifact must be traceable to a single business transaction.



The canonical root identifier is:



```text

businessTransactionId

```



All trust-chain artifacts derive their lineage from this identifier.



\---



\# Lineage Hierarchy



```text

businessTransactionId

&#x20;       │

&#x20;       ▼

&#x20;   decisionId

&#x20;       │

&#x20;       ▼

&#x20;    receiptId

```



Supporting lineage identifiers:



```text

subjectId (optional)

```



\---



\# Lineage Identifiers



\## businessTransactionId



\### Purpose



Represents the business operation being evaluated.



Examples:



```text

PAYMENT-001



TRANSFER-9821



REFUND-771

```



\### Characteristics



```text

Required



Business-generated



Stable



Cross-system identifier



Root trust-chain identifier

```



\### Scope



A single business transaction may produce:



```text

Multiple signals



Multiple decisions



Multiple attestations



Multiple receipts



Multiple executions

```



All artifacts remain linked through the same businessTransactionId.



\---



\## decisionId



\### Purpose



Represents a specific authority evaluation outcome.



Example:



```text

d5e9a3c1-fcc4-4a7d-ae5f-c7f873f55f88

```



\### Characteristics



```text

Required



Globally unique



Immutable



Generated by Parmana

```



\### Relationship



```text

businessTransactionId

&#x20;      │

&#x20;      └── decisionId

```



A transaction may have multiple decisions over time.



Examples:



```text

PAYMENT-001



├─ decision A

└─ decision B

```



Decision history must be preserved.



\---



\## receiptId



\### Purpose



Represents verification of a decision attestation.



Example:



```text

r7b88b56-4b27-4d58-9c8f-4c44d2f94b63

```



\### Characteristics



```text

Required



Globally unique



Immutable



Generated after verification

```



\### Relationship



```text

decisionId

&#x20;     │

&#x20;     └── receiptId

```



A decision may have multiple verification receipts.



Examples:



```text

decision A



├─ receipt 1

├─ receipt 2

└─ receipt 3

```



Verification history must be preserved.



\---



\## executionId



\### Purpose



Represents execution performed by an external system.



Example:



```text

e851b9c6-c7fb-439c-aedb-c8486734dc85

```



\### Characteristics



```text

Required



Globally unique



Immutable



Generated during execution recording

```



\### Relationship



```text

receiptId

&#x20;     │

&#x20;     └── executionId

```



An execution record references the receipt used to authorize execution.

\# Supporting Identifiers



\## taskId



Represents the action under evaluation.



Examples:



```text

payment.release



refund.issue



transfer.execute

```



Characteristics:



```text

Required



Stable



Business-defined

```



\---



\## policyId



Represents the governing policy.



Examples:



```text

payment-policy-v1



refund-policy-v2

```



Characteristics:



```text

Required



Versioned



Organization-defined

```



\---



\## policyVersion



Represents the exact policy version used during evaluation.



Examples:



```text

1.0.0



2.3.1

```



Characteristics:



```text

Required



Immutable



Evaluation-specific

```



\---



\## subjectId



Represents the subject associated with the transaction.



Examples:



```text

employee-123



customer-987



vendor-444

```



Characteristics:



```text

Optional



Business-defined



May be absent

```



Subject identity is not required for trust-chain validity.



Cryptographic lineage must not depend on the presence of subjectId.

\# Lineage Relationships



\## Signal Evidence



```text

businessTransactionId

&#x20;       │

&#x20;       └── Signal Evidence

```



Signals are linked directly to the transaction.



\---



\## Authority Decision



```text

businessTransactionId

&#x20;       │

&#x20;       └── decisionId

```



Decisions reference:



```text

taskId



policyId



policyVersion

```



\---



\## Decision Attestation



```text

businessTransactionId

&#x20;       │

&#x20;       └── decisionId

```



Attestations cryptographically bind:



```text

businessTransactionId



decisionId



taskId



policyId



policyVersion

```



\---



\## Verification Receipt



```text

decisionId

&#x20;     │

&#x20;     └── receiptId

```



Receipts prove successful attestation verification.



\---



\## Execution Record



```text

receiptId

&#x20;     │

&#x20;     └── executionId

```



Execution records prove external execution occurred.

\# Lineage Invariants



\## LIN-001



Every trust-chain artifact must reference a lineage identifier.



\---



\## LIN-002



businessTransactionId is the canonical root identifier.



\---



\## LIN-003



decisionId must uniquely identify a decision.



\---



\## LIN-004



receiptId must uniquely identify a verification receipt.



\---



\## LIN-005



executionId must uniquely identify an execution record.



\---



\## LIN-006



Lineage identifiers are immutable after creation.



\---



\## LIN-007



Trust chains must be reconstructable using businessTransactionId alone.



\---



\# Architectural Principle



Parmana does not execute actions.



Parmana preserves verifiable lineage connecting:



```text

Signals

&#x20;   ↓

Decision

&#x20;   ↓

Attestation

&#x20;   ↓

Receipt

&#x20;   ↓

Execution

```



The lineage model provides the identifiers that make that trust chain auditable, replayable, and independently verifiable.
