# Parmana Trust Core

## Verification Receipt & Transparency Log Architecture

### Version 0.1

---

# Overview

Parmana ensures that only human-authorized actions are executed in AI systems.

The Trust Core provides cryptographic verification, evidence generation, receipt persistence, and transparency logging for authority verification workflows.

This document describes the first operational implementation of Parmana's evidence architecture.

---

# Architectural Principle

Humans define authority.

Parmana makes authority verifiable and enforceable before execution.

AI systems may generate requests, recommendations, or actions, but authority remains external to the AI system.

Parmana evaluates trusted signals against organizational policy and produces verifiable evidence before execution occurs.

# Components

## 1. Decision Attestation

A Decision Attestation represents evidence that an action has been authorized according to a trust profile.

Example fields:

```text
decisionId
evidence
signatures
metadata
outcome
```

Attestations are generated by authority-bearing systems and submitted to Parmana for verification.

---

## 2. Trust Policy

Trust Policies define the authority requirements necessary for approval.

Example:

```json
{
  "name": "default",
  "requiredAlgorithms": [
    "ed25519"
  ],
  "minimumSignatures": 1
}
```

Policies determine:

* Required signature algorithms
* Minimum signature thresholds
* Verification criteria

---

## 3. Verification Engine

The verifier evaluates attestations against policy requirements.

Verification produces:

```text
valid
verifiedAlgorithms
failedAlgorithms
```

Verification does not execute actions.

Verification only determines whether authority requirements have been satisfied.

---

## 4. Verification Receipt

A Verification Receipt is the canonical evidence artifact produced by Parmana.

Structure:

```text
receiptId
decisionId
valid
verifiedAlgorithms
failedAlgorithms
verifiedAt
```

Example:

```json
{
  "receiptId": "85bbdbf1-3eda-48da-89c6-77631b5869bf",
  "decisionId": "test-decision-001",
  "valid": true
}
```

---

# Receipt Persistence

Verification receipts are persisted in Supabase.

Table:

```sql
verification_receipts
```

Columns:

```text
receipt_id
decision_id
valid
verified_algorithms
failed_algorithms
verified_at
receipt_hash
```

Receipts can be independently retrieved using:

```http
GET /receipt/{receiptId}
```

This enables auditability and evidence retention.

---

# Receipt Hashing

Each verification receipt is hashed using SHA-256.

Algorithm:

```text
SHA-256
```

Purpose:

* Tamper detection
* Immutable receipt identity
* Transparency log integration
* Merkle tree compatibility

Example:

```text
3c31382599460d6185d1f4bf07762ca922f01beeb0901b89c5c8d1b6f0385fdd
```

Receipt hashes are persisted alongside verification receipts.

---

# Transparency Log

Every verified receipt generates a transparency log entry.

Table:

```sql
transparency_log
```

Columns:

```text
id
receipt_id
receipt_hash
created_at
```

Transparency logs provide:

* Independent evidence tracking
* Tamper-evident audit records
* Future inclusion proofs
* Merkle root generation

# Alignment with Parmana

Parmana does not execute actions.

Parmana verifies authority.

Execution remains the responsibility of the underlying system of record.

Canonical responsibility model:

```text
AI captures requests
        ↓
Systems provide facts
        ↓
Parmana verifies authority
        ↓
Execution system acts
```

Organizations decide what to trust.

Parmana ensures those trusted signals satisfy policy before execution.
